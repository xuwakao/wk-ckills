(This numbering matches SKILL.md RULE 1–7 exactly. Rules 0 and 0b are global principles applicable across all RULEs.)

0. PRODUCTION QUALITY: Do NOT default to "simplest correct." Choose performant data structures/algorithms (not naive O(n²)), handle edge cases and failure modes, write production-ready code. Simplicity is not an excuse for poor performance or missing error handling.

0b. TOKEN EFFICIENCY: Grep/Glob before Read. Read diffs not full files on re-review. Cite [F-NNN]/[plan#section] instead of re-quoting. Don't re-summarize state already in banner/docs. Truncate non-load-bearing output in investigation logs. Avoid redundant reads within a phase. Use Explore subagent for broad codebase searches. Prefer tables over prose. D.7 Retrospective records /usage output and identifies expensive patterns. Warning signs: re-reading unchanged files, >2 INCONCLUSIVE diagnosis rounds (prefer BLOCKED), overly broad subagent prompts.

0c. ANTI-GUESSING — EVIDENCE OR MARKER: Every technical claim (about codebase, library, API, syscall, performance) MUST either cite inline evidence — [src/file.rs:Lstart-Lend], [verified: <command>], [docs.rs/...URL], [A# CONFIRMED], [POC: ...] — OR be tagged [GUESS] with the specific verification action that will resolve it. [GUESS] cannot be used as basis for plan structure, design choice, root cause, or PASS verdict. Resolve all [GUESS] before relying on them. Common guess hotspots: "how does this codebase work" (read it), "how does this library behave" (read its source), "standard way to do X" (search this codebase first), "should work on platform Y" (verify per platform), "fast/scalable" (benchmark, not reason), "bug is probably in X" (RULE 5b runtime evidence required).

RULE 1. SELF-MONITORING: Re-read the active plan file before each phase. If unsure about rules, re-read SKILL.md at ${CLAUDE_PLUGIN_ROOT}/skills/forge/SKILL.md. PreToolUse hook injects this summary every 10 tool calls (warn-only; never blocks). Stop hook blocks Claude self-stopping while plan is ACTIVE.

RULE 2. CONTINUOUS EXECUTION: Do not pause, do not ask "shall I continue?", do not explain next steps — execute them. SIX legitimate stop reasons (no others): (1) user interrupt (ESC/Ctrl+C); (2) user text pause request → mark plan PAUSED; (3) unsolvable issue → RULE 3 re-plan trigger; (4) A.0 Open Questions — STOP and ask user for ambiguous requirements; (5) D.5 User Acceptance Gate — STOP and wait for user accept/reject; (6) RULE 5b Stop Condition — after 2 investigation iterations no CONFIRMED root cause, mark BLOCKED and report to user. Stops 4/5/6 are mandatory correctness, not deviations.

RULE 3. PHASED DEVELOPMENT: A.0 → A.1 → A.2 → A.3 → A.4 → A.5 → A.6 → A.7 → B → C (loop) → D → RULE 7. Strict order, no skipping.
   - A.0 REQUIREMENTS: create plan file from template. Produce "## Requirements" with User Stories, Acceptance Criteria (AC-1, AC-2... business-level + testable by outsider), NFRs. Ambiguous? STOP + ask via Open Questions. Do NOT invent requirements.
   - A.1 CODEBASE RECONNAISSANCE: produce "## Codebase Reconnaissance" with S# search actions table (real Grep/Read/Glob outputs), files read with what each contributed, existing patterns identified, similar features (use as reference), constraints discovered. "I read the code" without this artifact = A.1 skipped.
   - A.2 ARCHITECTURE DECISION (ADR): Stage 0 PRIOR ART SURVEY (mandatory) — "### Prior Art Survey" with P# rows from real Grep/Read/WebSearch/WebFetch across this codebase + dependencies + comparable projects + authoritative specs + known pitfalls. THEN list alternatives (each referencing P#). Produce ADR-NNN (codebase-wide sequential), Context/Alternatives/Decision/Consequences/Supersedes. Status PROPOSED → ACCEPTED after A.3 passes.
   - A.3 FEASIBILITY RESEARCH: enumerate technical assumptions, verify each with RUNTIME/SOURCE-LEVEL action (cargo tree, grep dep source, man page, curl, minimal POC). Record A1/A2… in "## Feasibility Research" table with CONFIRMED/REJECTED/INCONCLUSIVE. High-risk → 10-50 line POC, run it. ANY REJECTED/INCONCLUSIVE → plan blocked.
   - A.4 phase definitions: each phase traces to AC-N (AC coverage); deps/risks/related ADRs.
   - A.5 TEST PLAN per phase via INDEPENDENT QA SUBAGENT (inputs = AC+NFR+Phase headings only, NOT ADR/code). Unit/Integration/E2E cases. Main writes subagent output verbatim.
   - A.6 finalize (verify all sections present, Status=ACTIVE). A.7 create progress.
   - META-PHASE B Plan Review: MULTI-ROUND table with columns Dependencies/ExpectedResults/Feasibility(cite A#)/Risks/Stub-Real/AC-Coverage/Test-Plan-Adequate. Plus cross-cutting: dependency graph (no cycles), alternatives completeness, AC completeness (every AC covered), NFR coverage. FAIL/RISK → fix plan → re-review "(round M)".
   - META-PHASE C per phase: C.1 Pre-Phase (re-read plan, restate expected results). C.2 Execute (record findings). C.3 by INDEPENDENT SUBAGENT: C.3a Outcome + C.3b Code (Investigation Log first, then Table with [investigation:...] refs). UNLIMITED ROUNDS until zero FAIL/CONCERN. C.4 Functional Acceptance (RULE 4). C.5 mark COMPLETE (pre-condition: both review artifacts + acceptance present).
   - Plan corrections during execution are append-only (mark old [DEPRECATED], create new, cross-reference).

RULE 4. FUNCTIONAL ACCEPTANCE: 4.1 Build (fail→STOP). 4.2 Execute every Test Plan test by ID (unit/integration/E2E) with command+output. 4.3 Regression (run full existing test suite). 4.4 NFR verification (or explicit deferral to D). Planned test not executed = violation.

RULE 5. DEBUGGING DISCIPLINE: No workarounds (5a: no unwrap/todo/unreachable as fixes, no silent catches). TWO-STAGE DIAGNOSIS (5b): Stage 1 INVESTIGATE first (read failing code + RUNTIME TOOLS: lldb/gdb backtrace, real log excerpts, strace/dtrace, ThreadSanitizer, profiler, instrumentation with reproduction). Reading code alone is NOT debugging — it produces hypotheses, not verifications. Stage 2 hypotheses grounded in investigation; verification methods RUNTIME-level. Root cause requires CONFIRMED from runtime evidence. STOP CONDITION: after 2 iterations no CONFIRMED → BLOCKED, report to user. 5c: pick tool by failure class. 5d: fix process = RULE 3+4 (plan→review→execute→code-review→acceptance). 5e escalation: 3 failed fixes→change strategy; 5 failed plans→re-plan; N similar issues→broader systemic check.

RULE 6. DOCUMENTATION + KNOWLEDGE: Update docs/progress/ and docs/issue/ continuously. Formal technical language only. [UNVERIFIED] for unverified. /git-commit for commits. STRUCTURED FINDINGS (Type: DECISION/DISCOVERY/CONSTRAINT/WARNING/BENCHMARK/GAP, with Statement/Evidence/Impact/Tags/Status — not bare titles). Empty Findings = skipped recording. PROJECT-LEVEL KNOWLEDGE under docs/knowledge/: api/ (user-facing API docs — update in same phase that changes public API), onboarding/ (README/architecture/conventions/setup), ownership.md (module matrix — update when modules change), runbooks/ (operational procedures). ADR (A.2, plan-level one-per-plan) vs DECISION finding (implementation-level, many-per-plan): different scopes. D.1 audits all knowledge artifacts for staleness.

D.5 USER ACCEPTANCE GATE — STOP and wait for user "accept" or rework list. D.6 mark COMPLETED (only after user accepts). D.7 Retrospective (What went well / didn't / Root causes / Process improvements / DECISION findings promoted to ADR / /usage token review). D.8 → trigger Autonomous Planning.
   - AUTONOMOUS: scan docs/ for IN-PROGRESS/BLOCKED issues, [UNVERIFIED] findings, PENDING phases, GAP findings. If any → new plan → return to A.0. Only stop if nothing remains.
